# GitHub Integration

Visual Hive emits GitHub-ready markdown instead of requiring privileged API calls in the default workflow.

Generated artifacts:

- `.visual-hive/report.json`
- `.visual-hive/baselines.json`
- `.visual-hive/mutation-report.json`
- `.visual-hive/issue.md`
- `.visual-hive/pr-comment.md`
- `.visual-hive/triage-prompt.md`

Recommended PR permissions:

```yaml
permissions:
  contents: read
```

`pr-comment.md` is a sanitized comment body draft. Post it only from a workflow with the permissions your project intentionally grants; the default templates keep PR validation read-only and only upload artifacts.

Issue creation should happen in a trusted follow-up workflow that consumes uploaded artifacts. Do not create issues directly from untrusted PR code with write permissions or secrets. The trusted workflow generated by `visual-hive init` downloads artifacts, recursively locates `issue.md` regardless of whether `.visual-hive` was uploaded as the artifact root or as a nested directory, redacts secret-like values again, and dedupes issues from deterministic failure evidence instead of the unique workflow run ID.

Use `pull_request`, not `pull_request_target`, for untrusted PR validation.

Run `visual-hive workflows` to write `.visual-hive/workflows.json`, which audits actual workflow YAML for PR secret usage, write permissions, `pull_request_target`, hidden artifact upload, baseline review queue generation, and trusted `workflow_run` issue patterns. Trusted issue workflow checks include artifact download, recursive `issue.md` discovery, dedupe markers, no checkout, and a defensive redaction pass before issue creation.

Run the workflow audit before `visual-hive triage` when you want `.visual-hive/issue.md` and `.visual-hive/pr-comment.md` to include workflow-safety evidence:

```bash
visual-hive workflows
visual-hive baselines list --write
visual-hive triage
visual-hive report --github-step-summary
```

The workflows generated by `visual-hive init` run the baseline review queue and workflow audit before triage and before uploading `.visual-hive`, so the trusted follow-up lane can inspect baseline decisions, workflow safety evidence, and deterministic reports together.

The same built-in PR, scheduled, and trusted failure issue templates are exposed in the Control Plane GitHub / CI page as copyable snippets and guarded write actions. The UI writes only the built-in template paths under `.github/workflows`, refuses accidental overwrites unless explicitly forced after review, honors `--read-only`, and records `.visual-hive/workflow-edits.json`. The CLI and UI read the shared core template registry, so `visual-hive init` and the local Control Plane present the same safe defaults.

The CLI can also write the same guarded templates without starting the UI:

```bash
visual-hive workflows --write-templates
visual-hive workflows --write-templates --template pull_request
visual-hive workflows --write-templates --template trusted_failure_issue --force
```

`--write-templates` is the explicit confirmation gate. Existing workflow template files are not overwritten unless `--force` is passed after reviewing the diff or current file content. Every write records `.visual-hive/workflow-edits.json` and then writes `.visual-hive/workflows.json` so the setup PR contains both generated files and workflow safety evidence.

The GitHub adapter redacts token-like and secret-like values from issue and comment bodies.
