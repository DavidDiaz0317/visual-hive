# GitHub Integration

Visual Hive emits GitHub-ready markdown instead of requiring privileged API calls in the default workflow.

Generated artifacts:

- `.visual-hive/report.json`
- `.visual-hive/mutation-report.json`
- `.visual-hive/issue.md`
- `.visual-hive/pr-comment.md`
- `.visual-hive/triage-prompt.md`

Recommended PR permissions:

```yaml
permissions:
  contents: read
```

`pr-comment.md` is a sanitized comment body draft. Post it only from a workflow with the permissions your project intentionally grants; the default templates keep PR validation read-only and only upload artifacts.

Issue creation should happen in a trusted follow-up workflow that consumes uploaded artifacts. Do not create issues directly from untrusted PR code with write permissions or secrets.

Use `pull_request`, not `pull_request_target`, for untrusted PR validation.

Run `visual-hive workflows` to write `.visual-hive/workflows.json`, which audits actual workflow YAML for PR secret usage, write permissions, `pull_request_target`, hidden artifact upload, and trusted `workflow_run` issue patterns.

Run the workflow audit before `visual-hive triage` when you want `.visual-hive/issue.md` and `.visual-hive/pr-comment.md` to include workflow-safety evidence:

```bash
visual-hive workflows
visual-hive triage
visual-hive report --github-step-summary
```

The workflows generated by `visual-hive init` run this audit before triage and before uploading `.visual-hive`, so the trusted follow-up lane can inspect workflow safety evidence alongside deterministic reports.

The GitHub adapter redacts token-like and secret-like values from issue and comment bodies.
